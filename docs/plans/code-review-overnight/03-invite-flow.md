# Review 03: invite-issuer.ts + tokens/invite.ts

Scope: `packages/auth/src/invite-issuer.ts`, `packages/auth/src/tokens/invite.ts`. Cross-referenced callers: `apps/web/app/auth/_lib/materialize-invite.ts`, `apps/web/app/auth/(default)/invite/{start/route.ts,actions.ts}`, `apps/web/app/onboarding/member/{actions.ts,_lib/invite-cookie.ts}`, `apps/web/app/onboarding/actions.ts`, `packages/auth/scripts/issue-invite-token.ts`, `packages/db/migrations/0002_auth.sql`, `packages/db/src/schema/auth_invite.ts`.

## Critical (BLOCKER)

- **[apps/web/app/onboarding/member/actions.ts:66-90 + 113-125] — invite redemption skips email-match check when a session already exists (auth bypass).** `submitMemberPasswordAction` calls `auth.api.getSession()` and, if a session is present, **skips `signUpEmail` entirely** and calls `materializeInvite(...)` against `claims.email` / `claims.organizationId` / `claims.role` for whatever `userId` the existing session points to. There is no `if (session.user.email !== claims.email) reject` guard anywhere on this path. The sibling action `acceptInviteAction` (`apps/web/app/auth/(default)/invite/actions.ts:70-75`) does enforce this check — the member-flow does not. Materialize itself also does not verify the email against the user's app_user row (see `materialize-invite.ts:59-185`). Attack: alice@evil.com signs in normally, intercepts/forwards bob@target.com's invite link, hits `/auth/invite/start?token=…` (cookie set), then walks the new-member wizard to `/onboarding/member/password`. The "idempotency guard" detects alice's pre-existing session, skips signUpEmail, and materializes a workspace_membership + organization_membership pinning alice's user_id to bob's invited organization at the invited role. The invite is then marked accepted, locking bob out and granting alice cross-tenant access.
  Fix (defense-in-depth at both layers): in `submitMemberPasswordAction`, after obtaining `session`, before calling `materializeInvite`, assert:
  ```ts
  if (session?.user?.email?.toLowerCase() !== claims.email.toLowerCase()) {
    return { ok: false, errorKey: "inviteEmailMismatch" }
  }
  ```
  And in `materializeInvite`, look up `app_user.email` for `input.userId` and compare against `input.email` inside the transaction so admin scripts / future callers cannot regress this invariant either.

- **[packages/auth/src/invite-issuer.ts:203-209] — case-sensitive email comparison silently bypasses revocation, allowing concurrent valid tokens for the same recipient.** `revokePendingInvites` builds the WHERE as `eq(auth_invite.email, input.email)`. The DB trigger `app_auth_invite_email_normalize` (migrations/0002_auth.sql:230-243) lowercases `auth_invite.email` on INSERT/UPDATE, so existing rows are always lowercase. If the caller passes a mixed-case email (and `InviteRowSchema` in `packages/shared/src/auth/onboarding.ts:67-74` does not lowercase — Zod `.email()` accepts mixed-case as-is), the equality fails to match the stored row. `submitTeamAction` (`onboarding/actions.ts:436-451`) then calls `issueInvite` immediately after, the old pending row is **never** revoked, and the recipient ends up with two simultaneously redeemable tokens. The comment on `revokePendingInvites` ("Call this BEFORE issuing a fresh invite so an older token can no longer be redeemed") is silently wrong on this input shape. The same casing bug masks the unique-token invariant — both rows have distinct `token_hash`, so the `UNIQUE` constraint on `token_hash` is no help.
  Fix: normalize email at the boundary before any DB comparison:
  ```ts
  export async function revokePendingInvites(input: {
    organizationId: string
    email: string
  }): Promise<number> {
    const email = input.email.trim().toLowerCase()
    // ... use `email` in the WHERE
  }
  ```
  And mirror the normalization in `issueInvite` (line 100) so the URL / email body / audit row are consistent with the DB write. Lowercasing in the Zod schema (`z.string().email().toLowerCase()`) is a second line of defense; do both.

- **[packages/auth/scripts/issue-invite-token.ts:26, 31, 38, 54] — broken import: `InviteClaims` no longer exists.** The file imports `import type { InviteClaims } from "../src/tokens/invite"` and uses it as `InviteClaims["role"]` in four places. `packages/auth/src/tokens/invite.ts` exports `InviteRecord` (and only that — the JWT-era `InviteClaims` symbol was removed). `tsc --noEmit` against this script fails immediately. Because the script gates on `APP_TOKEN_SECRET` (lines 93-99) and presents itself as the canonical "dev CLI mirrors the onboarding path" (header comment lines 2-7), it has likely never been run since the JWT→opaque-token migration landed. A dev attempting to re-issue an invite to debug a customer ticket cannot run this script today.
  Fix: replace the import with `InviteRecord` and re-type the role union:
  ```ts
  import type { InviteRecord } from "../src/tokens/invite"
  type InviteRole = InviteRecord["role"]
  // ... InviteRole everywhere
  ```
  Also note the file header comment (lines 2-7) still says "sign the JWT" which is no longer what `issueInvite` does — update the docstring or it will mislead the next reader.

## Warning

- **[packages/auth/src/invite-issuer.ts:30-45 + 53-133] — `issueInvite` and `revokePendingInvites` are not composed atomically; race window leaves prior invite redeemable.** `submitTeamAction` (`onboarding/actions.ts:439-450`) calls `revokePendingInvites(...)` and then `issueInvite(...)` as two separate `withAdminBypass` transactions. Between those calls, the recipient could redeem the older (still-pending) invite because the revoke commits before the new row is inserted. Worse, if the second call fails (DB hiccup, email transport down), the revocation is committed but no new invite exists — the recipient now sees their invite as "disabled" and the admin has no fresh link. The "issue then revoke previous" pattern is the standard fix; or wrap both in a single `withAdminBypass(outerTx => ...)` transaction.
  Fix: take a single `withAdminBypass` and pass the outer tx into a `revokePendingInvitesTx` helper, then INSERT the new row in the same transaction. Side benefit: also makes `auth_invite_organization_email_idx` lookups go through a single READ COMMITTED snapshot, eliminating the partial-state window.

- **[packages/auth/src/invite-issuer.ts:62-65] — raw invite token transmitted in URL query string; no Referrer-Policy header.** `url = ${baseUrl}/auth/invite/start?token=<32B raw>`. The token is the credential; anything that logs Referer (CDN access logs, intermediate proxies, third-party scripts loaded after the redirect) sees the full token. The route does set the cookie and 302 to `/auth/invite` (which drops the token from the URL bar), but: (a) email previewer fetchers (Gmail's image proxy, Outlook safelink rewriters, corporate email security gateways) hit the URL with the token in plaintext before the user does, and (b) any 3xx-chain in front of `/auth/invite/start` (an auth proxy, edge middleware) records the token. There is no project-wide `Referrer-Policy: no-referrer` or `strict-origin` header anywhere in the repo (grep returns 0 matches for `Referrer-Policy`).
  Fix at two layers: (1) add `Referrer-Policy: no-referrer` in the Next.js middleware/response headers for all `/auth/invite/*` routes (or globally — it's almost always the safer default); (2) consider POST-ing the token from `/auth/invite/start` into a server action that sets the cookie + redirects, removing the token from the URL entirely. Also document the token-rotation policy if a token is suspected to have leaked (status='revoked' is already supported — just confirm it's the recommended response in the runbook).

- **[packages/auth/src/invite-issuer.ts:150-188] — `readInviteByRawToken` returns the FULL record (incl. organizationId + role + email) for ANY token state, including 'revoked' and 'accepted'.** Callers in `route.ts` and `actions.ts` use it to render distinct error UIs, but the function also leaks the invited email + organization_id + role to anyone who holds the raw token, regardless of whether the invite is usable. A token that was revoked an hour ago still discloses "yes, bob@target.com was invited to organization X as admin." That is enumerable PII leakage given an attacker who guessed/intercepted a single token. The doc comment (lines 144-148) explicitly says "Returns null if the token does not exist (treated identically to 'expired' to avoid token-enumeration leaks)" — which acknowledges the threat model — but then returns the full claims for revoked/accepted/expired statuses anyway.
  Fix: split into two functions: `readUsableInvite(rawToken)` returns the full record only for `status='pending' && !expired`, otherwise returns a discriminated `{ status: 'expired' | 'revoked' | 'accepted' | 'not-found' }` with no claims. Callers in the route handler don't need the email/organizationId/role to render the error page anyway — they only need the status string.

- **[packages/auth/src/invite-issuer.ts:62-63 + 154 + apps/web/app/auth/_lib/materialize-invite.ts:62, 187-189] — hash lookup is not constant-time.** `hashInviteToken` is plain `sha256(rawToken).hex`; `eq(auth_invite.token_hash, tokenHash)` uses a B-tree index. Equality comparison on the hex hash inside Postgres is byte-by-byte and short-circuits — a timing-side-channel attacker measuring response time on `/auth/invite/start` can in theory recover hash prefix bits. The hash space is 256 bits and the token entropy is 256 bits, so practical exploitation is not realistic for an external attacker against a server hop, but the pattern is worth fixing: store and compare a hash that has a uniform-time equality (e.g., compare with `bytea` and `digest()` rather than hex `text`), or accept the residual risk and document it. Lower priority than the items above; flagging because the file's header comment claims "industry-standard … design used by Stripe / GitHub / Linear" and those teams use HMAC of token under a separate key, not bare SHA-256.
  Fix: optional — switch from `sha256(token)` to `hmac-sha256(token, INVITE_HASH_KEY)` with a separate key (`INVITE_HASH_KEY`, distinct from `APP_TOKEN_SECRET`). Limits damage if the `auth_invite` table dump leaks (an attacker who exfiltrates `token_hash` rows cannot brute-force tokens offline without also obtaining the HMAC key).

- **[apps/web/app/onboarding/actions.ts:431] — `baseUrl` falls back to `http://localhost:3000` silently in production.** `const baseUrl = process.env.BETTER_AUTH_URL ?? "http://localhost:3000"`. If `BETTER_AUTH_URL` is unset on a production deploy, every invite email contains a `http://localhost:3000/auth/invite/start?token=…` link. Recipients cannot accept; the audit row records the invite as sent but the link is unreachable; admin sees no error. This is also a usability landmine in staging environments. Same gap exists in `packages/auth/scripts/issue-invite-token.ts:57`.
  Fix: in production (`NODE_ENV === 'production'`) require `BETTER_AUTH_URL` and throw at module load if missing. In dev, keep the localhost fallback. Mirror the `server.ts` validation pattern that the previous review already flagged.

- **[packages/auth/src/invite-issuer.ts:78, 109, 153] — error messages leak organization-id existence + DB internals to callers.** Throws like `Organization ${organizationId} not found` and `auth_invite insert returned no row` propagate up to `submitTeamAction` (`onboarding/actions.ts:452-458`) which puts the literal `err.message` into `failures[].reason` — and `submitTeamAction`'s return value flows to the client. That means an admin browsing the team-step UI can probe "does organization with id `X` exist on this deploy" by submitting an invite with a guessed organization id (which they should not even be able to choose, but if they could via direct action invocation). Lower-severity than the auth bypass above because the action does already scope `defaultOrgId` server-side, but the pattern of pushing internal error text to the client is worth fixing.
  Fix: catch on the server, log full error with `console.error`/`logger.error`, return a flat `"saveInviteFailed"` errorKey to the client. The pattern already exists for other onboarding steps; team-step diverges.

- **[apps/web/app/auth/(default)/invite/start/route.ts:18-58] — no rate-limit / brute-force protection on the token-lookup endpoint.** `GET /auth/invite/start?token=<X>` is anonymous, requires no captcha, no rate limit, no IP gate. An attacker can probe ~10^6 random tokens/second from a single IP. The token entropy (256 bits, base64url 43 chars) makes guessing astronomically unlikely, so this is information-disclosure-only — but combined with the `readInviteByRawToken` claims-leak above, a single guessed token would disclose org+email+role for the lifetime of the row (including 30 days after status='accepted'). Adding rate-limiting eliminates the timing-side-channel concern in the previous warning too.
  Fix: add Better Auth's rate-limit config (or the project's middleware-level rate limiter) for `/auth/invite/start` — e.g. 30 req/min/IP. Document in the auth runbook.

## Info / Quality

- **[packages/auth/src/invite-issuer.ts:9, 30, 32, 33, 47] — multiple module-level concerns mixed in one file.** `issueInvite`, `readInviteByRawToken`, `revokePendingInvites`, `findOrganizationOwner`, `expireDuePendingInvites` are five different responsibilities (write, read, mutate-batch, lookup-helper, scheduled-task). The file is 258 lines and the `findOrganizationOwner` function is unrelated to invites — it queries `organization_membership` for owners and only exists because the CLI script and team-step both want it. Move `findOrganizationOwner` to a dedicated `organization-helpers.ts`; the import surface in `apps/web/app/auth/_lib/issue-invite.ts:7` already proves it's a separate concern.

- **[packages/auth/src/invite-issuer.ts:174] — comparison uses `<=` where the migration uses `<`.** Line 174 `if (status === "pending" && row.expires_at <= new Date()) status = "expired"`. The DB job `expireDuePendingInvites` (line 251) uses `WHERE expires_at < now()`. Materialize uses `${auth_invite.expires_at} > now()` (so a row with exactly-now expires_at is **not** acceptable — equivalent to `<= now()` is expired). The three call sites are inconsistent at the boundary `expires_at == now()`. Pick one — the conventional choice is `expires_at <= now()` is expired everywhere — and align all three to the same predicate. Practical impact is microscopic (sub-second window) but consistency matters for tests + audit reasoning.

- **[packages/auth/src/invite-issuer.ts:246-258] — raw SQL string with `unknown as Array<…>` cast.** `await db.execute(sql\`UPDATE ... RETURNING id\`) as unknown as Array<{ id: string }>` bypasses Drizzle's typed query builder. The same UPDATE is expressible with `db.update(auth_invite).set({ status: 'expired' }).where(and(lt(auth_invite.expires_at, sql\`now()\`), eq(auth_invite.status, 'pending'))).returning({ id: auth_invite.id })` and gets a real `Array<{ id: string }>` return type without the `unknown` escape. Fewer drift surfaces.

- **[packages/auth/src/tokens/invite.ts:40-53] — `InviteRecord` interface duplicates the schema's typed columns.** `role: "owner" | "admin" | "member" | "agent" | "guest"` is hand-maintained here AND in `packages/shared/src/auth/onboarding.ts:73` (`z.enum(["admin", "member"])` — note the difference: shared/onboarding restricts to admin|member, invite.ts widens to all five). When `auth_invite.role` is `varchar(64)` not enum (intentional per schema comment), the union is purely an app-layer convention. If a future migration adds `'analyst'` to the enum, every union literal must be updated. Centralize: derive `InviteRole` once from a shared `auth_invite_role` Zod enum and import it everywhere (`packages/shared/src/auth/*.ts` and re-export from `invite.ts`).

- **[apps/web/app/auth/_lib/materialize-invite.ts:113-126] — defense-in-depth check `org.id !== inviteRow.organization_id` is unreachable.** The function selects `organization WHERE id = input.organizationId` (line 116-118), then compares `org.id !== inviteRow.organization_id` (line 121). The two values are only equal-compared because the caller (acceptInviteAction line 79) passes `record.organizationId` (which IS `auth_invite.organization_id`) into `input.organizationId`. The branch never fires unless a caller passes a mismatched organizationId — which would itself be a bug. The comment "JWT claim must match the row's organization_id" is stale (JWT path no longer exists). Either delete the dead branch or repurpose it: pull `organizationId` from the auth_invite row directly (via the UPDATE's `RETURNING organization_id`) and drop the parameter, eliminating the class of callers-pass-wrong-thing bugs.

- **[packages/auth/src/tokens/invite.ts:29-30 + 32-34] — unused export.** `INVITE_TOKEN_BYTES = 32` is exported but no consumer references it (`grep INVITE_TOKEN_BYTES` returns only the declaration + the re-export in `tokens/index.ts:15`). Either drop the export or add a test that asserts `generateRawInviteToken().length === expectedLengthFor(INVITE_TOKEN_BYTES)` so changes to the constant fail loudly.

- **[packages/auth/src/invite-issuer.ts:17-27] — comment block is partly stale.** "Caller should `revokePendingInvites()` first when re-issuing for the same (organization, email)" — true today, but the BLOCKER on race-window above suggests the better contract is "issueInvite itself revokes prior pending invites for the same (org, email)." If that refactor happens, this comment goes stale; if not, hoist the contract into a single helper function `replacePendingInvite()` that does both in one transaction and update the comment to point at it.

- **[packages/auth/src/invite-issuer.ts:223-238] — `findOrganizationOwner` returns the FIRST owner.** A single org can have multiple `organization_membership` rows with `role='owner'`. Using `.limit(1)` without `ORDER BY` returns whichever happens to be cheapest for the planner. For the CLI's audit field (`issued_by_user_id`), this is "any owner is fine" — but the behavior is non-deterministic. Add `.orderBy(organization_membership.created_at)` so reruns return the same owner and the audit trail attributes consistently.

- **[packages/auth/scripts/issue-invite-token.ts:67-72] — UUID regex permits any 8-4-4-4-12 hex string, not validating UUID version/variant.** `^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i`. Postgres' `uuid` type accepts it, so functionally fine; but the script's CLI failure message says "is not a valid UUID" which is misleading for malformed inputs that happen to match the shape. Use a stricter `^[0-9a-f]{8}-[0-9a-f]{4}-[1-7][0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i` if you want to enforce v1-v7, or just let Postgres reject it on the insert and surface the DB error.

- **[apps/web/app/onboarding/member/_lib/invite-cookie.ts:34-38] — `readRawInviteToken` is a footgun.** It returns the raw credential as a string with no usage guard. Callers must remember NOT to log it / pass it to client-facing surfaces. Wrap it in a branded type `RawInviteToken` (zero-runtime newtype) and have `materializeInvite` accept that type, so accidental string concatenation into a log line fails at compile time.

---

## Severity Summary

- **BLOCKER (3)**: cross-tenant invite acceptance via existing session (member onboarding skips email match); case-sensitive revocation silently leaves prior invite redeemable; dev CLI broken at type-check (stale `InviteClaims` import).
- **WARNING (7)**: revoke-then-issue not atomic; raw token in URL + no Referrer-Policy; `readInviteByRawToken` leaks claims for non-pending statuses; SHA-256 of raw token (no HMAC key separation); `baseUrl` fallback to localhost in prod; internal error messages leaked to client; no rate-limit on the token-lookup endpoint.
- **INFO (8)**: file mixes five concerns; `<` vs `<=` inconsistency at expiry boundary; raw SQL with `unknown` cast; duplicated role union; unreachable defense-in-depth check; unused export; stale comments; non-deterministic owner lookup; UUID regex; `readRawInviteToken` newtype.

The opaque-token + DB-hash architecture is sound (correct improvement over the JWT-era design). The bugs are all at the boundary: input normalization, caller contract enforcement, and consistency between the two redemption paths (`acceptInviteAction` vs `submitMemberPasswordAction`). The single most important fix is the email-match check in the member-onboarding action.
