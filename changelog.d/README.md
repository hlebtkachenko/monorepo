# changelog.d/ â€” changelog fragments

The pending (unreleased) changelog lives here as one small Markdown file per
change, instead of a shared `## [Unreleased]` block in `CHANGELOG.md`. Unique
filenames mean parallel PRs never edit the same file, so they never conflict.
At release-cut the files are folded into a new `CHANGELOG.md` version section
and deleted. `CHANGELOG.md` is written **only** at release-cut; do not edit it
by hand.

See the whole pending release at any time:

```bash
pnpm changelog:preview
```

## The rule: one fragment per commit

Write the fragment in the **same commit** as the change it describes:

```bash
pnpm changelog:add -- --category Fixed --entry "One-sentence description of the change"
git add changelog.d/ <changed files>
git commit -m "fix(web): ..."
```

Squash away `wip` / `fixup` commits before they reach `main`, so every commit
that lands carries exactly one fragment = one changelog bullet. A PR that makes
three distinct changes has three commits and three fragments. The CI gate
requires at least one fragment per non-release PR (Dependabot and
`chore(release): vX.Y.Z` PRs are exempt).

## A fragment

```markdown
---
category: Fixed
---

Org switcher now preserves the current module and page when switching
organizations, instead of returning to the organization root.
```

The **body** (everything under the closing `---`) IS the changelog line â€” the
exact bullet that lands in `CHANGELOG.md`. Write it as one complete,
user-readable sentence; that is what a reviewer reads in the PR diff and what a
reader sees in the release. There is no separate title field.

Frontmatter:

| Key        | Required | Values                            | Meaning                                                                                                                                                   |
| ---------- | -------- | --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `category` | yes      | one of the sections below         | Which `### Section` the bullet renders under.                                                                                                             |
| `bump`     | no       | `patch` (default) `minor` `major` | The version-bump level this change implies. The strongest across all fragments is the release's suggested bump. A level, never a concrete version number. |
| `override` | no       | `true`                            | The `bump` is deliberate â€” the release step honors it without re-deriving against a rule.                                                                 |

Categories (Keep a Changelog, already used across `CHANGELOG.md`):
`Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`, `Docs`,
`Dependencies`.

Filenames are `<name>-<hex>.md`, auto-generated by `changelog:add`; the hex
suffix guarantees uniqueness. Do not edit or delete another PR's fragment.

## Automatic vs manual

| Step                                                                                     | Who                                        |
| ---------------------------------------------------------------------------------------- | ------------------------------------------ |
| Write a fragment per commit                                                              | you / agent (`changelog:add`)              |
| Gate: PR added â‰Ą1 valid fragment                                                         | automatic (CI `check`, pre-push)           |
| Assemble `CHANGELOG.md` + backfill `(#PR)` + fold in Dependabot bumps + delete fragments | automatic (`changelog:collect` at release) |
| Choose the version number and push the tag                                               | manual (release owner)                     |
